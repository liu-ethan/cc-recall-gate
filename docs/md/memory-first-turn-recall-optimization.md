# Claude Code 记忆系统优化：意图/置信度门控的首轮召回

> 定位：这不是“给 Agent 加一个向量数据库”，而是在主模型前增加 Memory Recall Gate，通过“明确需要则同步、不确定则异步、明确不需要则跳过”的三路决策修复记忆召回与 Agent Loop 之间的时序缺口。

## 1. 一句话定位

本优化位于“用户 Prompt → 组装上下文 → 调主模型”之间：Memory Recall Gate 与记忆预取并发启动；Gate 判断当前请求对历史记忆的依赖程度后，明确需要记忆时等待已经运行的预取并在**本 Turn 第一次 `callModel` 前**注入，不确定时保持异步，明确不需要时取消预取并跳过注入。

### 1.1 术语对齐（必读）

学习与面试时必须先对齐三个词，否则会把“会话第一句”和“一句对话里的第一次模型调用”混为一谈：

| 口语 | 系统语义 | 例子 |
|------|----------|------|
| **会话（Session）** | 新开一个标签 / 一次 REPL 会话 | 关掉标签再开 = 新会话 |
| **对话 / Turn** | 用户每一次提问（一次 User Turn） | 同一标签里连续问三句 = 三个 Turn |
| **首轮（本文）** | **同一 Turn 内**第一次调用主模型（尚未进入 Tool Loop） | 不是“会话里的第一句对话” |

因此：本优化修的是 **每个 Turn 内部的时序缺口**，不是“只在新会话第一句注入记忆”。相关记忆预取在代码里是 **每个用户 Turn 启动一次**（`query.ts`）。

## 2. 为什么需要它

### 2.1 第一性原理

记忆系统的目标不是“存下更多内容”，而是让正确的信息在模型做决定时可用：

```text
记忆有效价值 = 信息正确性 × 决策时可用性
             - 召回延迟 - Token 噪声 - 陈旧信息风险
```

如果相关记忆在模型已经完成回答后才返回，那么无论召回结果多准确，本轮的“决策时可用性”都是 0。

### 2.2 三层记忆模型：各注入什么、何时写入

Claude Code 的“记忆”不是一个模块，而是三种生命周期不同的机制。三者**不要理解成“每轮一起塞进 prompt”**：

```text
Auto Memory     → 跨会话长期知识（索引常驻 + 详细文件按 Turn 召回）
Session Memory  → 本会话滚动笔记 summary.md（对话中写，Compact 时读）
Compact/Budget  → 窗口治理（不负责跨会话回忆）
```

#### 2.2.1 Auto Memory（跨会话）

**存什么**：用户画像、反馈、项目背景、外部资源指针；`MEMORY.md` 索引 + 主题文件（`user / feedback / project / reference`）。

**怎么进上下文（拆成两路）**：

| 通路 | 时机 | 是否贯穿每个 Turn |
|------|------|-------------------|
| **索引 `MEMORY.md`（及 CLAUDE.md 栈）** | Session 开始经 `getUserContext` 加载并会话内缓存 | 是：该会话每次 API 调用都能看到 |
| **详细主题文件（`relevant_memories`）** | **每个 User Turn** 异步预取；通常在 Tool 执行后消费注入 | 否：按句召回 + 去重，不是开局全量灌入 |

已具备能力：会话结束/轮末后台 Fork 抽取；Sonnet 相关性选择（最多 5 个文件）；单文件 4KB、单会话累计 60KB；已读去重、陈旧提示、失败降级。

启用相关性预取实验（`tengu_moth_copse`）后：**AutoMem/TeamMem 的 `MEMORY.md` 不再常驻注入**，更依赖按 Turn 的详细召回——因此不能把索引当成可靠兜底。

#### 2.2.2 Session Memory（本会话）

**存什么**：当前长会话的结构化笔记 `summary.md`（标题、Current State、任务、关键文件、错误、Worklog 等章节模板）。

**`summary.md` 怎么来的？（高频误解）**

- **不是**“固定前 n 轮对话的切片”。
- **不是**“只有 Compact 才写入”。
- **是**：主模型采样结束后的 `postSamplingHook` 里，按阈值周期性触发后台 Fork Agent，用 Edit **增量更新**同一份 `summary.md`。

写入触发（与 Compact 解耦，默认量级）：

```text
上下文约 ≥ 10k tokens  → 才允许第一次抽取（冷启动）
之后约再涨 5k tokens   → 才考虑再次更新
通常还需工具调用间隔（默认约 3 次 tool call）
  或「本轮无 tool 且 token 够了」的自然停顿
        ↓
Fork Agent 读当前 summary.md → Edit 各章节 → 写回
并用 lastSummarizedMessageId 标记“笔记已覆盖到哪条消息”
```

**Compact 做什么**：主要是 **读** `summary.md`，用笔记替换旧消息、保留最近窗口；Compact 前可能等待进行中的抽取，但 Compact **不是**写入开关。

删掉 Session Memory 会坏在哪：超长会话 Compact 后失去“当前任务/已试错路径”的结构化连续性；它不解决跨标签回忆。

#### 2.2.3 Compact / Tool Result Budget

不是第三种“记忆库”，而是上下文窗口治理：限制历史消息与工具结果占用；Session Memory Compact 是压缩策略之一。本优化切口不在这一层。

因此，不应再把“增加长期记忆”“增加异步摘要”“接入向量库”作为项目主叙事。这些方案没有击中当前控制面的主要矛盾——**Auto Memory 详细召回与 Agent Loop 的时序缺口**。

### 2.3 当前控制流缺口

缺口只针对 **Auto Memory 的详细文件召回**，发生在**每一个**需要记忆、但可能无 Tool Call 的 User Turn：

```text
用户输入（任意第 N 句对话，不是仅会话第一句）
  ↓
启动详细记忆异步预取          ← 每 Turn 一次
  ↓
组装当前上下文
  ↓
第一次调用主模型                ← 详细记忆尚未注入（本文“首轮”）
  ↓
解析响应
  ├─ 没有 tool call → 直接结束   ← 预取结果本 Turn 不会被使用
  └─ 存在 tool call
       ↓
     权限校验 → 执行工具
       ↓
     消费记忆预取结果
       ↓
     写回消息 → 第二次调用模型   ← 到这里才看到详细记忆
```

这会影响最典型的记忆请求：

- “你还记得我之前说测试不要用 Mock 吗？”
- “按我之前喜欢的方式解释。”
- “上次讨论的合规约束是什么？”
- “还是按之前那个方案来。”

这些请求很可能直接产生文本回答，不会触发 Tool Loop；越需要记忆，越可能拿不到详细记忆。

删掉本优化会坏在哪里：需要跨会话信息、但本 Turn 第一次模型调用前不调用工具的请求仍然可能“失忆”，而现有召回延迟指标即使很好，也无法证明记忆真正参与了模型决策。

## 3. 核心设计

### 3.1 总体方案

引入一个轻量的 `Memory Recall Gate`（代码层可抽象为 `RecallPolicy`），形成三路决策、两条召回通路。为了避免“先等 Gate、再启动记忆检索”产生两段串行延迟，Gate 和现有记忆预取应当同时启动：

```text
用户 Prompt
   ├──────────────────┐
   ↓                  ↓
启动 Memory Gate    启动记忆预取
   │                  │
   └────── 等待 Gate 决策 ──────┐
                                ↓
   ├─ SYNC_RECALL  → 等待同一个预取 Promise
   │                  → 注入首轮上下文 → 调主模型
   ├─ ASYNC_RECALL → 不等待预取
   │                  → 先调主模型 → 后续 Loop 按需注入
   └─ SKIP_RECALL  → 取消预取
                      → 直接调主模型
```

这是一种“投机预取”：`SYNC_RECALL` 可以隐藏一部分检索延迟，`ASYNC_RECALL` 完全复用当前路径，`SKIP_RECALL` 虽然可能已产生少量预取开销，但应立即取消。三条分支必须共享同一个预取句柄，不能同步、异步各发一次请求。

其中“用户要求忽略记忆”必须归入 `SKIP_RECALL`。核心不是用同步召回替换异步召回，而是根据当前决策是否依赖记忆，选择是否在主模型前等待已经启动的召回结果。

### 3.2 RecallPolicy 的输入与输出

输入只使用语义信息，不要求学生背字段：

- 当前真实用户请求；
- 少量最近对话，用于理解“那个方案”之类的省略表达；
- 记忆文件的 filename、type、description、mtime；
- 当前会话已注入或已读取的记忆集合；
- 用户是否要求忽略记忆。

Gate 可以由轻量模型实现，输入 Prompt 后输出三类决策：

```text
SYNC_RECALL：明确需要记忆，同步等待召回并在本 Turn 第一次 callModel 前注入
ASYNC_RECALL：是否需要记忆不确定，继续现有异步预取（默认安全区）
SKIP_RECALL：明确不需要记忆，取消预取且不消费结果
```

轻量模型只负责判断召回策略，不负责回答原问题。它的输出必须是受约束的枚举，分类失败或超时时降级为 `ASYNC_RECALL`，避免阻塞整个 Agent Loop。Gate 的超时与记忆预取的超时应分别记录，否则无法判断延迟来自分类还是检索。

#### ASYNC vs SKIP 怎么划界（第一性原理）

不问“要不要记忆”的程度，而问：**本 Turn 若现在不注入，最坏坏在哪？预取成本值不值？**

| 决策 | 语义 | 代价偏好 |
|------|------|----------|
| `ASYNC_RECALL` | **不确定**是否需要 | 宁可多花一点预取，保留后续 Loop 回灌期权 |
| `SKIP_RECALL` | **明确不需要**（或用户禁止） | 宁可漏召回，也不付 Token / 延迟 / 噪声 |

进 `SKIP_RECALL` 的充分条件：用户显式忽略记忆；请求语义自洽、无跨会话依赖与省略指代；注入记忆几乎不可能改变本轮决策。

进 `ASYNC_RECALL` 的充分条件：普通编码任务（偏好有时有用但不硬依赖）；弱指代但不够硬；**Gate 失败/超时**（安全默认必须是 ASYNC，不是 SKIP）。

口诀：负例要保守——只有“明确无关 / 用户禁止”才 SKIP；中间地带全部 ASYNC。

### 3.3 同步召回通路

Gate 返回 `SYNC_RECALL` 后，等待与 Gate 并发启动的记忆预取 Promise，完成记忆选择和文件读取，再组装首次 `messagesForQuery`，最后调用主模型。第一版不引入向量数据库，只在最多 200 个 frontmatter 上完成候选选择：

- filename 与 query 的匹配；
- description 与 query 的匹配；
- memory type 与意图匹配，例如用户偏好优先匹配 `feedback`；
- 字符 n-gram 或基础关键词分数，兼容中英文；
- mtime 作为弱特征，而不是把旧记忆直接淘汰。

只注入 Top 1～2，并复用现有单文件 4KB 限制。这样首轮最多额外注入约 8KB 原始文本，不会无界占用上下文。无论候选选择使用本地排序还是现有 selector，“同步”的系统语义都不变：主模型第一次调用前必须拿到召回结果或明确的空结果。

如果 Gate 判断“明确需要记忆”，但同步召回超时、失败或没有匹配结果，不能让主模型假装记得。应在首轮上下文中加入“没有找到可用记忆”的受控 meta 提示，让模型明确说明无法确认历史信息，随后正常继续任务。

### 3.4 慢通路

Gate 返回 `ASYNC_RECALL` 后，继续复用现有 Sonnet selector：

- 不阻塞普通请求的首 Token；
- 最多选择 5 个候选；
- 在后续 Tool Loop 中消费；
- 通过 `readFileState` 和 surfaced paths 与同步召回通路去重。

如果首轮没有 Tool Call，异步结果仍然可能来不及消费，这是该分支主动接受的正确性代价；Gate 的目标是把强记忆依赖请求尽量路由到 `SYNC_RECALL`，而不是让异步路径保证所有请求都能召回。

### 3.5 本 Turn 第一次 callModel 前的注入点

同步召回结果必须在第一次 `callModel` 前并入 `messagesForQuery`，不能继续放在工具执行后的 attachment 收集阶段。这里的“第一次”指 **当前 User Turn 内**，与会话是否刚开无关。

设计上应保持：

- 记忆作为尾部 meta user message/attachment 注入，不修改稳定的 system prompt 前缀；
- 选择结果写入 transcript，恢复会话时保持上下文可解释；
- Gate 超时降级到异步；同步召回超时或失败时注入“记忆不可用”提示，不允许模型伪造回忆；
- 用户要求忽略记忆时短路所有召回路径。

### 3.6 MVP 边界

第一版只完成一个闭环：

1. 增加轻量 Memory Recall Gate，输出 `SYNC_RECALL / ASYNC_RECALL / SKIP_RECALL`；
2. Gate 与现有记忆预取并发启动，三条分支共享同一个 Promise 和 AbortController；
3. `SYNC_RECALL`：等待预取，读取 Top 1～2，并在第一次主模型请求前注入；
4. `ASYNC_RECALL`：不等待预取，保留后续 Loop 回灌；
5. `SKIP_RECALL`：取消预取，完全跳过本轮记忆召回；
6. Gate 失败或超时时降级到异步路径，同步召回失败时禁止模型伪造回忆；
7. 建立控制流测试和离线回放评测。

暂不做 embedding、向量数据库、自动遗忘、团队记忆同步或新的摘要模型，避免同时铺开多个方向。

## 4. 关键 tradeoff

### 4.1 正确性 vs TTFT

始终等待模型重排能提高语义召回，但会让所有请求增加首 Token 延迟。门控方案只对明确依赖历史的请求支付同步成本；不确定请求继续异步，明确无关请求直接跳过。

轻量 Gate 本身也会成为所有请求的串行开销，因此必须单独衡量 Gate 的 p50/p95 延迟。若这部分成本不可接受，可以进一步把规则快判放在模型 Gate 前，或把“是否需要召回”和“选择哪些记忆”合并为一次轻量模型调用，但这属于同一方案的性能优化，不改变三路控制流。

失败模式：意图判断漏掉隐式回忆请求。应通过离线回放统计门控 Recall，而不是不断增加关键词规则。

### 4.2 召回率 vs 上下文噪声

首轮注入越多，Recall@K 可能越高，但错误记忆和无关偏好也更容易干扰模型。首轮只注入 Top 1～2，低置信候选留给慢通路。

### 4.3 本地检索 vs 向量检索

本地 frontmatter 排序延迟低、可解释、没有索引一致性问题，但语义能力有限。当前只有最多 200 个候选，而且 description 本来就是为相关性判断设计的，第一版没有必要先引入向量数据库。

只有当评测证明 lexical/n-gram Recall 明显不足，才应把 embedding 作为后续实验变量。

### 4.4 新鲜度 vs 长期有效性

项目状态衰减较快，用户偏好可能长期不变。因此不能只按 mtime 排序：

- `project` 类型可以给 freshness 更高权重；
- `user / feedback` 类型应降低时间衰减；
- 涉及代码、文件和 Flag 的记忆在行动前仍需读取当前状态验证。

### 4.5 Prompt Cache 稳定性 vs 动态上下文

把记忆写入 system prompt 会破坏稳定前缀；把它放在当前用户请求之后的尾部 attachment，可以减少缓存影响。恢复会话时还需要复用已持久化的选择结果，避免同一历史随机选出不同记忆。

## 5. 和相邻模块的接口

### 上游：用户输入与会话上下文

契约：提供当前真实用户请求和必要的最近对话，不把 Hook、系统提醒等 meta message 当成召回意图。

### 当前模块：记忆扫描与选择

契约：

- 输入是查询语义与 memory manifest；
- 副作用只有读取记忆文件；
- 不需要用户权限确认，但仍受路径边界和 AbortSignal 约束；
- 输出是可注入的记忆 attachment 或空结果。

### 下游：Agent Loop 上下文组装

契约：`SYNC_RECALL` 结果在首次 `callModel` 前进入 `messagesForQuery`；`ASYNC_RECALL` 沿用工具执行后的回灌路径；`SKIP_RECALL` 不产生记忆 attachment。

### 去重与预算

契约：复用 `readFileState`、历史 `relevant_memories` attachments、单文件 4KB 和单会话 60KB 限制，不能建立第二套不一致的预算状态。

### Compact / Resume

契约：压缩清理旧 attachment 后允许重新召回；恢复会话时保留已注入记忆的 path、content、mtime/header，确保上下文可重放。

### 权限边界

该优化只读记忆文件，不执行写操作或外部 Tool。写入仍由现有主 Agent 或后台 extractMemories Agent 完成，不扩张权限面。

## 6. 面试高频问法与标准答纲

### Q1：为什么不能继续完全异步召回？

- **结论**：异步召回优化了延迟，但在无 Tool Call 的直接回答场景破坏了正确性。
- **机制**：当前预取结果只在工具执行后回灌；若首轮直接结束，详细记忆不会进入模型上下文。
- **Tradeoff**：不能因此改成所有请求同步等待，所以增加意图/置信度门控。

### Q2：为什么不用向量数据库？

- **结论**：当前候选规模和数据形态下，向量库不是最小必要方案。
- **机制**：候选最多 200 个，每个文件已有专门用于相关性选择的 description，本地排序可以覆盖大部分显式请求。
- **Tradeoff**：语义召回可能较弱，因此保留 Sonnet 慢通路，并用 Recall@K 决定是否需要 embedding。

### Q3：如何防止错误或陈旧记忆污染模型？

- **结论**：记忆是历史声明，不是当前事实源。
- **机制**：低置信不注入；涉及文件、函数和 Flag 的行动前重新检查当前状态；用户要求忽略记忆时直接短路。
- **Tradeoff**：会增加少量漏召回和工具验证成本，但换取更低的错误行动率。

### Q4：为什么首轮只注入 1～2 个文件？

- **结论**：首轮目标是支持关键决策，而不是最大化 Recall@5。
- **机制**：`SYNC_RECALL` 只注入高置信 Top 1～2，其余不确定请求走异步重排；同时复用 4KB/文件预算。
- **Tradeoff**：可能漏掉弱相关记忆，但能显著降低 Token 噪声和错误关联。

### Q5：怎么证明这个优化不是 Demo？

- **结论**：同时验证质量、时延、成本和错误率。
- **机制**：使用跨会话回放集，对比当前异步、全量阻塞、门控双通路三组方案。
- **Tradeoff**：不能只展示单个成功例子，也不能只用 LLM 主观打分；需要可复现标签和端到端结果。

## 8. 落地实施、测试与量化

本项目只做一个切口：**让相关记忆在首轮决策前可用，同时避免所有请求同步阻塞。** Runtime 改造、控制流测试、端到端指标不是三个优化点，而是同一个优化从“做出来”到“证明有效”的三个交付阶段：

```text
Runtime 改造 = 让 CC 真的按三路策略运行
控制流测试   = 证明超时、中断、去重等分支没有写错
端到端指标   = 证明改造后质量提升，延迟和 Token 成本可控
```

### 8.1 第一阶段：Runtime 改造

目标产物是可运行的功能代码，而不是只有设计图。按下面顺序实现：

1. 在用户 Prompt 进入主 Loop 后，同时启动轻量 Gate 和现有记忆预取；
2. Gate 输出受约束的 `SYNC_RECALL / ASYNC_RECALL / SKIP_RECALL`；
3. `SYNC_RECALL` 在首次 `callModel` 前等待预取并注入记忆；
4. `ASYNC_RECALL` 不等待，沿用工具执行后的消费点；
5. `SKIP_RECALL` 取消预取，不产生记忆 attachment；
6. 让三条分支共享同一个预取 Promise、AbortController 和消费状态；
7. 复用现有 `readFileState`、surfaced paths 和 Token 上限，防止重复注入；
8. Gate 超时降级为异步，召回失败不终止对话，用户中断能取消所有子请求；
9. `SYNC_RECALL` 没找到记忆时注入受控提示，禁止模型声称自己“记得”。

完成标准不是“代码能编译”，而是能够演示以下三条真实路径：

```text
明确回忆请求 → 首次API请求中已经包含目标记忆
不确定请求   → 主模型不等待，后续Tool Loop可消费记忆
无关请求     → 不发起或及时取消记忆召回
```

### 8.2 第二阶段：控制流测试

控制流测试不评价回答文案，而是通过 Mock Gate、Memory Prefetch 和 `callModel` 检查消息与状态是否按预期流转。

| 测试场景 | 需要验证的结果 |
|---|---|
| 首轮无 Tool Call，Gate=`SYNC_RECALL` | 第一次 `callModel` 的 `messagesForQuery` 已包含目标记忆 |
| 首轮有 Tool Call，但预取较慢 | 同步分支等待同一个预取任务，不重复发起召回 |
| Gate=`ASYNC_RECALL` | 第一次主模型调用不等待；预取完成后可在后续 Loop 注入 |
| Gate=`SKIP_RECALL` | 预取被取消，首轮和后续轮都没有记忆 attachment |
| Gate 超时或返回非法结果 | 自动降级到 `ASYNC_RECALL`，主 Loop 不被卡死 |
| 同步召回超时/失败/空结果 | 主模型收到“记忆不可用”约束，不伪造历史信息 |
| 同步和异步选中同一文件 | 通过 `readFileState` 只注入一次 |
| 用户按 Esc 中断 | Gate、预取和文件读取都收到 Abort，不残留后台注入 |
| 记忆超过预算 | 继续遵守单文件4KB、单会话60KB限制 |
| Compact / Resume 后继续对话 | 已持久化结果可重放，不出现重复或错误丢失 |

建议先写一个能够稳定复现旧问题的失败测试：构造“需要历史偏好、主模型直接文本回答、不产生 Tool Call”的请求，证明旧实现在第一次模型调用中没有详细记忆。随后实现新逻辑让该测试通过，这就是最清晰的改动证据。

### 8.3 第三阶段：端到端实验

端到端实验回答的不是“代码有没有走对分支”，而是“用户最终是否得到更正确、更快、成本更低的回答”。

准备一批固定回放样本，每条样本标注：

- 当前 Prompt 是否需要记忆；
- 正确的 Gate 分支；
- 应召回的目标记忆文件；
- 正确回答必须包含或遵守的事实；
- 不应该被注入的干扰记忆。

评测集至少覆盖：

- 明确回忆、且首轮不会调用工具的问题；
- 隐式偏好和“还是按之前那个”之类的省略表达；
- 普通代码问题等无关负例；
- 旧记忆与当前代码冲突；
- 用户明确要求忽略记忆；
- 相关记忆位于较旧文件；
- Gate 或记忆服务超时、失败的降级场景。

### 8.4 对照实验

使用完全相同的数据和模型配置运行三组：

- A：当前纯异步召回；
- B：所有请求都同步等待召回；
- C：Gate 与预取并行的三路门控方案。

A 组提供当前质量/延迟基线；B 组提供“质量优先但延迟最差”的上界；C 组验证门控是否接近 B 的正确性，同时保留接近 A 的普通请求延迟。

### 8.5 核心指标

| 指标 | 回答的问题 |
|---|---|
| Gate Accuracy / 分支混淆矩阵 | 需要、可选、不需要记忆的请求是否路由正确 |
| First-call Memory Availability | 第一次模型请求是否已包含目标记忆 |
| Recall@1 / Recall@5 | 目标记忆是否被选中 |
| Task Accuracy | 最终回答是否正确使用记忆 |
| False Injection Rate | 无关请求被错误注入记忆的比例 |
| Stale-memory Error Rate | 因陈旧记忆导致错误行动的比例 |
| Gate p50/p95 Latency | 轻量分类本身带来多少串行延迟 |
| Memory Recall p50/p95 Latency | 同步分支还需要等待多久 |
| p50/p95 TTFT Delta | 用户实际感知的首Token延迟变化 |
| Injected Tokens | 每轮额外注入多少 Token |
| Side-query Cost | Gate和召回增加或节省了多少调用成本 |
| Cancel / Timeout Success Rate | 中断和降级是否真正生效 |

最重要的三个验收指标是：

1. 显式回忆场景的 First-call Memory Availability；
2. 最终 Task Accuracy，而不只是“检索命中”；
3. 普通请求的 p95 TTFT Delta。

只有前两者提升且第三项受控，才能证明门控策略真正改善了“正确性 vs 延迟”的系统 tradeoff。

### 8.6 推荐执行顺序

```text
写失败测试，复现旧版首轮失忆
  ↓
实现 Gate 与预取并发启动
  ↓
接入 SYNC / ASYNC / SKIP 三路状态
  ↓
补齐超时、取消、空结果、去重和预算
  ↓
完成控制流自动化测试
  ↓
制作跨会话回放集
  ↓
运行 A/B/C 对照实验
  ↓
用真实数字写入简历
```

### 8.7 工作量与项目含金量判断

如果只增加一次轻量模型分类，再写一个 `if/else` 选择是否等待，改动确实偏小。面试官继续追问超时、取消、重复注入、预算和效果数据时，项目会迅速失去可信度。

本项目真正有价值的工作不是代码行数，而是把一个异步时序缺陷闭环解决：

1. **Runtime 语义完整**：改动首次模型调用前的上下文组装点，并处理共享 Promise、单次消费、Abort、超时降级和 Compact/Resume；
2. **边界可证明**：用控制流测试覆盖无 Tool Call、慢召回、重复候选和用户中断等失败模式；
3. **收益可量化**：用 A/B/C 回放实验说明首轮记忆正确性提升了多少，以及为此支付了多少 TTFT、Token 和调用成本。

因此，简历不要写“新增记忆意图分类器”，而应写“修复首轮记忆可用性的 Runtime 控制流，并完成正确性—延迟权衡评测”。前者像一个小功能，后者体现的是对 Agent Loop、异步并发、上下文生命周期和工程验证的完整理解。

## 结论

这个优化的核心价值不是“检索算法更高级”，而是修正 **Auto Memory 详细召回** 与 Agent Loop 的时序关系：明确需要历史信息时在本 Turn 第一次模型调用前同步注入，不确定时异步预取，明确不需要时跳过召回。

配套认知必须钉死三点，避免面试讲混：

1. **会话 ≠ 对话 ≠ 首轮**：Session / Turn / 第一次 `callModel` 是三层时间尺度。
2. **Auto Memory**：索引可会话常驻；详细文件按 Turn 召回——本优化只改后者的门控时序。
3. **Session Memory**：`summary.md` 在对话过程中按阈值写入，Compact 时读取；不是本优化切口。

它能够同时回答面试官最关心的三件事：

1. 控制面：记忆如何进入 Agent Loop；
2. Tradeoff：正确性、TTFT、Token 和陈旧风险如何平衡；
3. 可验证性：如何通过回放集和端到端指标证明优化有效。
