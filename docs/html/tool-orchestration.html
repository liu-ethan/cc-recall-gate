工具执行与编排

一轮里模型可能同时要读文件、搜代码、跑命令——Runtime 的工作不是「按顺序等」，而是决定谁能并行、结果何时回灌、失败怎么统一表达。

可并行（无依赖） 长耗时 / 瓶颈工具 按完成序流式回灌
先总体

编排层解决什么问题

协议定义「工具长什么样」；编排定义「一轮里多个工具怎么跑、结果怎么回到 Loop」。

工具编排 = 依赖感知的调度器 + 流式回灌通道 + 统一失败契约。

模型常在一个 turn 里发出多个 tool_use（读 A + 搜 B + 跑 C）。 若一律串行，总延迟是各工具之和；若盲目全并行，有写依赖时会竞态。 编排层的第一性原理：在正确性约束下最大化并发，并把结果尽早变成下一轮上下文。

采样 → 解析 tool_use → 权限 → 编排 · 执行 → 流式回灌 → 再采样
亮点 01

能并行就并行

分析工具间依赖：独立的同时启动，有数据依赖的串行，追求最大并发度。

亮点 02

流式回灌

不靠「全部完成再打包」；谁先结束谁先注入消息流，缩短空等。

亮点 03

统一错误 + 上限

失败形态归一成结构化 tool_result；并发有天花板，防止进程风暴。

后分点

三个设计亮点

每个亮点按「机制 → 删掉会坏在哪 → tradeoff」收束，方便面试口述。

01

「能并行就并行」的调度策略

不是按声明顺序串行等待，而是看依赖图再最大化并行。独立工具同时启动，总等待时间接近「最慢那条泳道」，而不是「所有泳道之和」。

  • 输入语义 本 turn 的一组 tool_use：各自要读什么、写什么、是否互斥（例如两个 Write 打同一文件）。
  • 调度规则 无数据依赖 → 并行；有依赖 / 冲突 → 串行或排队。目标是正确前提下的最大并发度。
  • 删掉会怎样 全串行：多工具 turn 延迟线性叠加，体感「Agent 很慢」；全盲目并行：写冲突、脏读，正确性崩。
Tradeoff 延迟 vs 正确性：并行度越高越快，但对依赖分析的要求越高。保守策略（宁可多串行）更安全，激进策略更吃依赖图质量。
02

流式回灌：边执行边注入

结果不是等所有工具跑完再一起发给模型，而是通过类似 StreamingToolExecutor 的通道——谁先完成，谁先变成 tool_result 消息。

  • 机制 工具在后台跑；完成事件触发结果写入对话流。模型侧看到的是按完成序到达的结构化结果，而不是「批次包」。
  • 收益 长耗时工具（如 Bash）不再拖死短工具的结果可见性；后续采样准备可以更早开始，降低端到端空等。
  • 删掉会怎样 等齐再回灌：最慢工具决定整批可见时刻，短工具的结果被无谓扣留，turn 延迟被人为拉长。
Tradeoff 吞吐 vs 批次一致性：流式回灌更快，但模型看到的结果顺序可能与 tool_use 声明顺序不同——协议层必须保证「用 tool_use_id 对齐」，而不是靠数组下标。
03

统一错误格式 + 并发上限

不管 Bash 报错、文件不存在还是超时，回给模型的都是同一种结构化 tool_result；同时用并发上限防止一次 turn 打爆本机进程。

  • 统一失败契约 错误是给模型看的控制信号，不是给人看的堆栈。格式统一，模型才能用同一套策略重试、换工具或向用户澄清。
  • 并发天花板 可并行 ≠ 无限并行。同时拉起过多 Bash/子进程会拖垮机器；编排层要有 in-flight 上限与排队。
  • 删掉会怎样 错误形态分裂 → 模型纠错不稳定；无上限 → 资源风暴，稳定性指标先崩，再谈智能。
Tradeoff 自主性 vs 稳定性：提高并发上限能缩短尾延迟，但抬高机器负载与失败率。面试口径：上限是可度量旋钮（p95 延迟、进程数、OOM 率），不是拍脑袋常数。

消化后能说什么

  1. 工具调度分析依赖关系：无依赖并行、有依赖串行，追求最大并发度。
  2. 结果不是等全部完成再发给模型，而是边执行边注入（流式回灌）。
  3. 所有工具错误用统一格式返回；并发有上限，防止同时启动过多危险进程。
