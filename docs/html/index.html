M1

主 Agent 与会话入口

★★★★★

学什么

Claude Code 从命令行启动到能接收用户输入的完整链路。main.tsx 是总入口，负责按顺序装配所有依赖（认证、遥测、配置、功能开关），然后启动 Ink 框架的 REPL 界面。REPL 收到用户输入后，交给 QueryEngine 作为会话门面，由它驱动整轮对话。QueryEngine 封装了「接收输入、调主循环、持久化、通知回调」的完整流程，所有外部调用者（REPL、Bridge、SDK）都通过这同一个入口发起对话。

设计亮点

  • 启动链的顺序装配：副作用从上到下、从前到后，每个阶段有明确依赖关系。不是事件驱动，而是顺序初始化，这让启动过程可预测、可调试。
  • QueryEngine 作为单一门面：所有外部调用者都通过 QueryEngine 进入主循环，而不是各自调用底层函数。这形成了清晰的 API 边界，换 REPL 框架或加新的入口（如 Web 端）都不影响核心。
  • 会话恢复不是「重放」而是「重建」：恢复时重建的是完整状态（消息列表、工具结果、权限决策），不是命令历史。这使得恢复后的会话可以无缝继续。

消化后能说什么

  • Claude Code 的启动是顺序依赖装配过程，所有副作用在启动阶段完成，核心循环是纯净的。
  • QueryEngine 是会话的门面，REPL/Bridge/SDK 都通过它进入主循环，而不是各自调用底层。
  • 会话持久化存的是状态快照，不是命令历史，所以恢复后可以无缝继续。
M2

Agent Loop 主循环

★★★★★

学什么

主循环是 Claude Code 的心脏。核心流程只有四步：采样（调用模型 API，传入消息历史和可用工具列表）→ 解析（从流式响应中增量解析 text 和 tool_use 块）→ 执行（调用对应工具的 execute 方法）→ 回灌（将工具结果作为 user/tool 消息注入对话流，触发下一次采样）。循环直到模型不再调用工具，或达到停止条件（max turns 耗尽、token 预算用光、用户中断、API 错误）。

设计亮点

  • Agent Loop 是通用模式：采样→执行→回灌→再采样，这个循环不是 Claude Code 特有的，而是所有 tool-use agent 的共性。理解这一个循环，就能理解所有 AI agent 的工作原理。
  • 流式处理的增量解析：不是等模型全部生成完再解析，而是边生成边解析。tool_use 的参数 JSON 可能在 chunk 边界被截断，解析器需要 pend 等待下一个 chunk 拼接完整。
  • 停止条件的多维度分层：有硬上限（max turns）、软上限（token budget）、用户中断、API 错误中止，每种停止的处理方式不同——有的优雅收尾，有的立即终止。

消化后能说什么

  • Agent Loop = 采样 → 工具执行 → 结果回灌 → 再采样，循环直到模型不再调用工具。
  • 流式处理的挑战在于增量解析：tool_use 参数可能在 chunk 边界被截断。
  • 停止条件是多维度的（turn 数、token 数、用户中断、API 错误），各有不同策略。
M3

会话记忆与上下文管理

★★★★★ 深入阅读 →

学什么

AI 的上下文窗口是有限的（200K tokens），但对话可能很长。这个模块负责三个核心问题：消息如何组织（user/assistant/tool_result/system 四类标准消息，所有对话状态通过消息流完全表达）；token 用量如何追踪（系统持续估算已用 token 数，输入+输出分开计算）；窗口快满时怎么压缩（compact 机制将历史对话压缩为紧凑摘要，保留结构信息如文件路径和行号，丢弃冗余细节）。系统提示词是分层组装的：base prompt 是静态的，工具列表每 turn 更新，用户自定义规则夹在中间。

设计亮点

  • 消息即状态：一份消息拷贝就是一份完整的会话状态快照。user→assistant（含 tool_use）→tool_result→assistant，整个对话的状态通过这个循环完全表达，不需要额外的状态变量。
  • Compact 的取舍艺术：什么该压缩、什么该保留直接影响 agent 的后续表现。策略是保留结构信息（文件路径、编辑行号）而压缩内容细节，让模型能记住「我改过哪些文件」而不是「文件里写了什么」。
  • Prompt 分层不是简单拼接：不同层有不同优先级和更新策略。工具列表每次 turn 更新，base prompt 静态不变，用户规则居间。每层独立管理，互不干扰。

消化后能说什么

  • 对话状态完全通过消息流表达：user→assistant(tool_use)→tool_result→assistant，以此循环。
  • Token 预算到顶时触发 compact：保留结构信息，压缩内容细节，释放窗口空间。
  • 系统提示词是分层组装的：base prompt + 工具列表 + 动态上下文，每层独立管理。
M4

Tool 协议定义

★★★★★ 深入阅读 →

学什么

所有工具（Bash、Read、Write、Glob、AgentTool、MCP 工具……）都遵循同一个接口协议。每个工具必须提供：name（唯一标识）、description（给模型看的自然语言说明）、inputSchema（JSON Schema 约束输入参数）、execute（核心执行逻辑，接收参数返回结果）、permission 钩子（是否需要用户确认）。模型通过 JSON Schema 理解工具的参数格式，不需要任何硬编码逻辑。新工具只需实现接口并注册，主循环代码不需要改动。

设计亮点

  • 工具即插件：所有工具是同一接口的不同实现。想加新能力？实现 Tool 接口即可，主循环代码完全不动。这是开闭原则的典范。
  • Schema 驱动：工具的能力声明（参数名、类型、是否必填、默认值）完全通过 JSON Schema 描述。模型通过 Schema 自己理解怎么用工具，不需要任何硬编码逻辑。
  • 协议不是实现：Tool.ts 定义的是「工具应该长什么样」，具体的 Bash、Read、Write 是「某个工具的长相」。这种分离让协议本身极其简洁，但能支撑几百种工具。

消化后能说什么

  • 所有工具遵循同一个接口：name + description + inputSchema + execute + permission。
  • 模型通过 JSON Schema 理解工具怎么用，不需要任何硬编码逻辑。
  • 工具即插件：想加新能力就实现 Tool 接口，主循环代码不需要改。
M5

工具执行与编排

★★★★☆ 深入阅读 →

学什么

模型可能在一个 turn 中请求多个工具（如「读文件 A + 搜索 B + 运行命令 C」）。系统需要决定哪些可以并行、哪些必须串行。核心策略是分析工具间的依赖关系：互相独立的并行执行，有数据依赖的串行执行。结果通过 StreamingToolExecutor 边执行边注入对话流——模型在处理前几个工具的结果时，后几个工具还在跑。所有工具错误用统一格式返回，让模型用一致的方式理解和响应。还有并发上限控制，防止同时启动过多 Bash 进程。

设计亮点

  • 「能并行就并行」的调度策略：不是按声明顺序串行等待，而是分析依赖图后最大化并行。独立的工具同时启动，大幅减少总等待时间。
  • 流式回灌：不是等所有工具都执行完再一起发给模型，而是边执行边注入结果。模型可以边看前几个工具的结果边继续思考后续行动。
  • 统一错误格式：不管什么类型的工具失败（Bash 报错、文件不存在、超时），返回给模型的错误信息格式统一，模型可以用一致的方式理解和响应。

消化后能说什么

  • 工具调度分析依赖关系：无依赖的并行，有依赖的串行，追求最大并发度。
  • 结果不是等全部完成再发给模型，而是边执行边注入（流式回灌）。
  • 所有工具错误用统一格式返回，模型看到的永远是结构化的 tool_result 消息。
M6

核心工具实现

★★★★☆ 深入阅读 →

学什么

挑选三个最典型的工具深入：Bash（子进程执行 shell 命令，捕获 stdout/stderr 和退出码，支持超时和目录设置，权限要求最高）；FileEdit（非破坏性的行级编辑：给定文件和行范围，原子替换，文件被外部修改时自动检测冲突并回退）；Grep（在代码库中搜索匹配内容，返回文件路径+行号+上下文，尊重 .gitignore）。它们代表三种副作用边界：进程执行（最危险）、写入磁盘（中等）、只读查询（安全）。

设计亮点

  • 三种副作用边界：进程（最危险）→ 磁盘写入（中等）→ 只读查询（安全）。权限模型根据边界分级，Bash 必须用户确认，Grep 可以自动执行。
  • FileEdit 的非破坏性设计：不是「覆盖整个文件」，而是「替换指定行」。如果行号在编辑前被外部修改了，编辑失败并提示冲突，不会静默覆盖。
  • 搜索工具的启发式价值：Grep + Glob 让 agent 能像人类开发者一样「先搜索再行动」，而不是盲目猜测文件位置。这是 agent 不盲目的关键。

消化后能说什么

  • 核心工具按副作用边界分为三级：进程执行 / 磁盘写入 / 只读查询，权限逐级递减。
  • FileEdit 是行级非破坏性编辑：替换指定行范围，文件被外部修改时自动检测冲突。
  • 搜索工具让 agent 能先探索代码库再行动，这是 agent 不盲目的关键。
M7

权限与人在回路

★★★★☆

学什么

Claude Code 能做的事很多（删文件、执行命令、发网络请求），不是所有操作都应该自动执行。权限系统提供 5 种模式：default（标准询问，危险操作弹窗确认）、accept-edits（自动接受编辑类操作）、auto（已信任的命令自动执行）、plan（只读模式，全部拒绝）、bypass（跳过所有确认）。每次工具执行前，useCanUseTool 钩子检查权限，返回允许/拒绝/询问。用户可以设置命令模式匹配规则（如「允许所有 git 命令」），并且每次决策会被记忆，下次相同命令不再询问，形成渐进式信任。

设计亮点

  • 人在回路：系统不会自作主张做危险操作。每次危险操作前暂停，等待人类判断。这不是限制，是安全保障。
  • 渐进式信任：从「全部询问」→用户逐步授权→越来越少的打断。信任是逐步建立的，不是一次性开关。
  • 可插拔的权限提供者：权限逻辑不是硬编码的，而是通过钩子注入。企业版可以替换为自己的权限策略。

消化后能说什么

  • 权限模式有 5 级：default→accept-edits→auto→plan→bypass，从最安全到最开放。
  • 每次工具执行前经过 useCanUseTool 钩子，返回 allow/deny/ask 三种决策。
  • 渐进式信任：用户逐步授权，系统记住决策，打断越来越少。
M8

多 Agent 编排

★★★★★

学什么

Claude Code 可以派生子代理独立完成任务。核心有两个概念：Agent-as-Tool——「派生子代理」本身被建模为一个 Tool 调用，子代理有独立上下文窗口，不受主代理 token 限制；Coordinator 模式——Commander 只有极少的工具（Agent、SendMessage、TaskStop），必须把工作委派出去，而 Worker 拥有完整工具集负责执行。Commander 负责拆解任务、编写自包含的 prompt、综合结果。Worker 看不到父对话的任何上下文，prompt 必须完全自包含。结果以 XML 格式注入主对话流。

设计亮点

  • Agent-as-Tool 是最优雅的抽象：把「创建子代理」建模为一个 Tool，而不是特殊的 API。子代理对主代理来说就像 Bash 或 Read 一样，只是一个工具调用。
  • Commander 的约束是设计的力量：Coordinator 没有 Bash、没有文件编辑工具——它必须把工作委派出去。这个约束不是限制，而是强制它扮演正确的角色。
  • Worker 看不到父对话：每个 Worker 的 prompt 必须完全自包含。这防止了上下文泄露，也迫使 Commander 给出精确指令——杜绝「甩锅式」委派。

消化后能说什么

  • Agent-as-Tool：派生子代理就是一个工具调用，子代理有独立上下文窗口。
  • Coordinator 不直接操作代码，只做拆任务→写 prompt→综合结果，工具集被刻意限制。
  • Worker 的 prompt 必须完全自包含，因为 Worker 看不到父对话的任何上下文。
