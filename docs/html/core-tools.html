亮点 1 · Process → Write → Read

三种副作用边界

机制

Bash 起子进程（最高危）→ FileEdit 写盘（中等）→ Grep/Glob 只读（安全）。权限随边界递减：危险操作必须确认，只读可自动执行。

Tradeoff

全自动快但不可控；全询问安全但打断多。按副作用分级，是在自主性与安全之间的可解释折中——而不是拍脑袋白名单。

面试答纲

结论：工具按副作用分级，权限跟着走。
机制：进程 / 写盘 / 只读三档。
代价：删掉分级 → 权限只能一刀切 → 要么烦死要么裸奔。

亮点 2 · Non-destructive edit

FileEdit 非破坏性行级编辑

机制

不是整文件覆盖，而是替换指定行范围并尽量原子提交。若文件在编辑前被外部改动，检测冲突并失败回退——不静默覆盖。

Tradeoff

换来的是正确性与可回退；代价是实现更复杂，吞吐不如一次 Write 整文件。Agent 场景里，静默损坏比多一次失败更贵。

面试答纲

结论：行级编辑 + 冲突检测，防静默损坏。
机制：按范围替换，外部变更则失败。
代价：为何不总用 Write？并发/外部修改下覆盖会丢人改的内容。

亮点 3 · Search before act

搜索启发式：先探索再行动

机制

Grep + Glob 让 agent 先定位再动手：返回路径、行号与上下文，并尊重 .gitignore。像人类开发者「先搜再改」，而不是猜路径硬闯。

Tradeoff

召回与上下文越完整，模型越不盲目；但结果会吃 token 与延迟。好的搜索工具要在「够用的上下文」和「不炸窗口」之间收束输出。

面试答纲

结论：搜索是 agent 不盲目的关键。
机制：Glob 找文件，Grep 找内容，带行号回灌。
代价：没有搜索 → 只能猜路径 → 盲目试错、浪费 turn。
